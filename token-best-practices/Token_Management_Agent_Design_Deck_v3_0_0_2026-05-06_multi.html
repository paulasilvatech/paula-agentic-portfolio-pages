01
Chapter 01 · Foundations

Token fundamentals.

The model never sees your text. It operates on integers indexed into a vocabulary of 32K to 200K entries. Without this foundation every cost estimate is a guess.

Practical equivalences

A token is not a word. It is a sub-word unit discovered by BPE on a training corpus.

Content type Words per token Why
English prose~0.75Common words map to single tokens; the densest case
Brazilian Portuguese~0.60Accents and cedillas split into multiple sub-words
Spanish~0.65Slightly denser than Portuguese
Python source~0.40Symbols, indentation, identifiers are token-heavy
TypeScript / JSX~0.35Generics and JSX symbols inflate further
Structured JSON~0.30Quotes, braces, separators dominate

Rule of thumb: a 1,000-line Python file ≈ 12,000 tokens. A 2,000-word PT-BR markdown ≈ 3,300 tokens. A 500-word EN response ≈ 670 tokens.

02
Chapter 02 · Foundations

Anatomy of consumption.

Tokens are not consumed uniformly. Three phases (input, output, cache), each with a different cost, hardware, and latency profile. Optimize each with different techniques.

Where the tokens go · empirical distribution

In real agentic SDLC workloads, input dominates and output is a rounding error.

The agentic multiplier

A 10-turn agent loop does not cost 10× a single call. It costs ~55×, input grows quadratically.

Turn 1
1×
Baseline input. System + tools + first user message.
Turn 5
15×
Conversation history + tool results accumulated four times over.
Turn 10
55×
Without compaction or caching, the loop re-sends everything every turn.

This is why "agent for everything" decks always blow their budget. Caching and compaction collapse this curve back toward linear.

03
Chapter 03 · Foundations

Models and multipliers.

Model is the first-order cost lever. Sonnet vs. Opus on the same task can be 5× without 5× more quality. Bundled vs. premium can be an order of magnitude.

Anthropic family · cost spread on identical workloads

Picking Sonnet over Opus by default is the single biggest lever most teams never pull.

Five-level heuristic

Default Sonnet. Step up to Opus only on demonstrable hard reasoning. Step down to Haiku when shape is fixed.

Level 1 · Trivial

Bundled or Haiku

Code completions, format fixes, doc lookups. No reasoning needed.

Level 2 · Routine

Bundled (GPT-4.1, GPT-5 mini)

Edit mode, single-file refactors, test stubs. Latency matters more than depth.

Level 3 · Default

Sonnet 4.6

Deep code review, agent mode, multi-file work. The 80% case.

Level 4 · Hard

Opus 4.7 · scoped

Architecture, distributed debugging, security review. Plan-only, then hand off.

Level 5 · Long context

Gemini 3.1 Pro

Whole-codebase analysis, log forensics, >200K context with no surcharge.

Anti-pattern

Opus as default

Most expensive failure mode. 5× the bill, <1.1× the quality on routine tasks.

04
Chapter 04 · Context engineering

Context rot. The marginal-utility curve.

More context is not better. After a threshold, every extra token adds cost without adding signal, and starts subtracting quality. The golden rule: minimum signal-rich context, never maximum context.

Context rot · measured

Cost climbs linearly with noise. Quality holds, until it cliffs.

Noise level Quality score Cost / session Verdict
00.95$0.10Optimal
1×0.94$0.20Cost doubles, quality flat, already losing money
5×0.88$0.60Quality starts dropping
10×0.79$1.10Visible degradation
50×0.62$5.10Severe rot · agent forgets the goal

Levers: compaction (structured compression), tool-result clearing, retrieval-on-demand. Chapter 4 of the playbook covers each.

05
Chapter 05 · Technical disciplines

Prompt caching. The 90% lever.

In conversational workloads, prefix caching cuts input cost by up to 90% without changing one line of prompt. But it requires architectural discipline: stable prefix, dynamic suffix, cache_control on the right blocks.

Cold · warm · the prefix-hash mechanic

A single byte change in the prefix invalidates the whole cache. Stable first, dynamic last.

06
Chapter 06 · Technical disciplines

Agent design.

There is a hype bias: agents for everything. Workflows solve 70% of enterprise cases at a fraction of the cost. This chapter teaches you to distinguish, choose, and implement agents only when they earn the spend.

The distinction that matters

Workflow vs. agent. The choice changes cost, latency, and operational risk.

Workflow · deterministic

LLMs and tools orchestrated by code paths.

  • Known, stable sequence. The LLM does parts; the code decides the order.
  • Cost is predictable. Same N calls per execution, regardless of input.
  • Easy to instrument, debug, audit step by step.
  • Low and stable latency. Fits 70% of enterprise cases.
Agent · dynamic

LLM decides which tools, in which order, based on intermediate results.

  • Sequence varies per input. Cost is a distribution, not a number.
  • Pays for the open exploration. Quadratic input growth without compaction.
  • Worth it when the problem is genuinely open-ended and tools are first-class.
  • Demands stop criteria, budgets, telemetry. Otherwise it loops forever.

The decision heuristic: if you can sketch the steps on paper, use a workflow. If you genuinely cannot, an agent earns its keep.

Reference architecture · 5 layers

The augmented LLM, the orchestration plane, and four token-cost levers, one per layer.

The five Anthropic patterns

Compose these. Do not invent. Each has a known cost and risk profile.

01

Prompt chaining

Decompose into ordered steps. Cheapest, most auditable. Default for SDLC subroutines.

02

Routing

A cheap classifier sends each request to the right specialist. Massive saves on mixed traffic.

03

Parallelization

Fan out independent subtasks, reduce latency. Watch the multiplier on cache writes.

04

Orchestrator-workers

Plan with Opus, execute with Sonnet/Haiku. The single best premium-to-bundled bridge.

05

Evaluator-optimizer

Loop with a critic. Bounded iterations or it never stops. Pair with a budget.

Anti-pattern

Multi-agent without coordination

N agents, no shared memory, no contracts. Pays N× and contradicts itself. Don’t.

Chapter 07 · Token-efficient strategies

Less is more. The 700-token review prompt and the 50-token review prompt produce equivalent reviews.

Verbose · 700 tokens

"Review this code carefully, considering security implications, performance characteristics, architecture decisions, code style, naming conventions, error handling, edge cases, and please be thorough and detailed in your feedback. Format your response as..."

Pays $700 / M tokens just on instructions. Repeated every turn.

Compact · 50 tokens

"Senior reviewer. Flag security, perf, correctness. JSON: {issues:[{severity, file, line, fix}]}."

93% reduction. In tests, review quality is equivalent or better, the model has more attention budget for the actual code.

Pair with progressive disclosure: a small SKILL.md on every call, large checklists loaded only on demand via a load_section tool.

FinOps as a habit

FinOps waterfall · cumulative savings

In sequence, the four levers compound from $1.00 to $0.18 per equivalent session.

Order matters. Each lever stacks on the previous one. Skipping model routing first leaves the largest delta unclaimed.

Chapter 12 · Five applied patterns

Five patterns you can paste into production this week. Each cuts a measurable percentage of spend.

01

Dynamic routing by complexity

Static keyword heuristic dispatches to Haiku / Sonnet / Opus. Zero LLM cost on the routing decision.

02

Hierarchical cache · 3 layers

In-process LRU → Redis → Anthropic prefix cache. Hits absorb 60–80% of repeated traffic.

03

Subagent fan-out

Independent subtasks parallel on Haiku, synthesis on Sonnet. Latency down, cost flat.

04

Plan / Execute

Opus designs the plan once. Sonnet/Haiku execute the steps. The premium-to-bundled bridge.

05

Semantic cache with fallback

Embedding-similarity hits return canned answers. Strict threshold + LLM fallback on miss.
