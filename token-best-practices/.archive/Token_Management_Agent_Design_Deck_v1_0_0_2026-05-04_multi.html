Capítulo 01 · Fundamentos do token

Um token não é uma palavra. É a unidade que o modelo processa.

Modelos modernos usam Byte Pair Encoding. Vocabulário descoberto empiricamente, sub-palavra. Densidade varia por linguagem e formato.

JSON estruturado~0,30Separadores e aspas pesam
As três categorias

Input, output, cached. Perfis de custo distintos.

INPUT
Paralelo, memory-bound
1× · Sonnet $3/M
System prompt, tools, histórico, contexto. Atenção em paralelo, gargalo é memória GPU. O mais barato por token.
OUTPUT
Sequencial, compute-bound
5× · Sonnet $15/M
Cada token depende de todos os anteriores. Não paraleliza dentro de uma resposta. Tipicamente 3-5× mais caro que input.
CACHED
Estado reaproveitado
0,1× · Sonnet $0,30/M
KV cache da atenção armazenado em GPU. Read custa 10% do input. Write custa 1,25× a 2× para popular.
Atenção crítica
Em modelos com extended thinking (Opus 4.7, GPT-5 com reasoning_effort), tokens de raciocínio invisíveis também contam como output. Uma resposta de 500 tokens visíveis pode ter consumido 5.000 de raciocínio interno.

O efeito mais subestimado

Em sessões agênticas, input cresce quadraticamente.

A matemática

Cada turno relê o histórico inteiro

  • Turno T usa input cumulativo de todos os turnos anteriores
  • Soma total: N×T + K×T(T-1) · termo dominante T²
  • Sessão de 20 turnos custa ~$7,44 sem cache
  • Mesmo agente com cache 85%: ~$3,57 (52% economia)
As alavancas

Como domar o multiplicador

  • Prompt caching · alavanca primária, 0,1× input em ~85% do contexto
  • Compaction estruturada · 22-57% redução, 0% perda de acurácia
  • Tool result clearing · libera 50K+ tokens em sessões longas
  • Subagent fan-out · 70-90% redução em workflows de exploração
Hierarquia de cache · 3 camadas

Combinadas: hit rate 60-90%. Custo de input cai 50-80%.

Capítulo 04 · Context engineering

A regra de ouro. Mínimo, não máximo.

Anthropic · Effective context engineering for AI agents · 2026

"O menor conjunto de tokens de alto sinal que maximiza a probabilidade do resultado desejado."

Não é "todo contexto possível". Não é "o que pareceu relevante à primeira vista". É o mínimo que efetivamente sustenta a probabilidade do resultado correto.

Tool result clearing

A maior fonte de tokens recorrentes. E a mais ignorada.

Um único read_file em arquivo de 2.000 linhas injeta 30K tokens. Dez files lidos em uma sessão = 300K só de tool results, todos pagos a cada turno subsequente.

A heurística
Manter os últimos 5-10 tool results é o sweet spot. Marcar como critical=true os resultados referenciados continuamente. Substituir tool results removidos por markers minimal: [Read file src/main.py: 1234 lines].

Em sessões longas, tool result clearing libera 50K+ tokens de um único turno. Aplique sempre que sessões passem de 10 turns.

Três artefatos. Especificação como código.

Capítulo 05 · Prompt caching

A alavanca de 90%. Prefix-stable + suffix-dynamic.

[1. System Prompt]      5K-50K tokens · estável · cache_control
   ├── Instruções gerais
   ├── Persona, role
   └── Documentação de longo prazo

[2. Tools definition]   1K-10K tokens · estável · cache_control
   └── Definições JSON de ferramentas

[3. Static context]     5K-100K tokens · opcional · cache_control
   ├── Documentos anexados
   └── Few-shot examples

[4. Conversation history]   dinâmico (cresce a cada turno)
[5. Current message]        dinâmico  ← NUNCA aqui o cache_control

Mínimo: 1.024 tokens (Sonnet/Opus) ou 2.048 tokens (Haiku). Anthropic suporta até 20 blocos com cache_control simultaneamente.

Capítulo 06 · Design de agentes

Workflow vs agente. A decisão arquitetural.

Workflow · 70% dos casos

Sequência de código pré-definida

  • Sequência conhecida e estável
  • Custo previsível, latência baixa
  • Fácil de instrumentar e auditar
  • Default. Comece sempre aqui.
Agente · 30% dos casos

LLM dirige seu próprio processo

  • Sequência decidida pelo modelo
  • Custo variável (10× ou mais)
  • Difícil de instrumentar e debugar
  • Adicione agência só quando workflow comprovadamente não dá conta
Anthropic · Building effective agents

Cinco padrões. Componíveis, não frameworks.

01
Prompt Chaining
Sequência fixa, output A vira input B. Linear, previsível, debugável.
02
Routing
Classifier direciona para pipeline especializado. Reduz custo médio 50-70%.
03
Parallelization
Sectioning (partes independentes) ou voting (robustez via múltiplos modelos).
04
Orchestrator-Workers
Lead decompõe dinamicamente, despacha. O padrão mais próximo de "agente verdadeiro".
05
Evaluator-Optimizer
Generator + avaliador em loop, com stop criteria robustos. TDD agent é o exemplo canônico.
Custom agents · .agents/

Single-responsibility. Read-only por default.

Accountability e controles

Showback primeiro. Chargeback após maturidade.

Accountability

Mês 1-6 · showback · mês 7+ · chargeback gradual

  • Showback · mostra custo sem cobrar (cria visibilidade)
  • Chargeback · debita orçamento da equipe (cria responsabilidade)
  • Pré-requisito · tagging consistente (<10% untagged)
  • Gradual · 25% → 50% → 75% → 100% mês a mês
Controles · 4 níveis

Do mais suave ao mais invasivo

  • Budget alerts (50%, 75%, 90%) · sempre · linha base
  • Budget caps · bloqueia premium, mantém bundled funcionando
  • Hard limits · raros · POC com budget fixo
  • Anomaly detection · 3σ · catch outliers proativo
