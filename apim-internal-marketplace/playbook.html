1. The sprawlA dispersãoLa dispersión

Modern enterprise development environments suffer from a specific and compounding form of sprawl. APIs proliferate without a central inventory. Shadow APIs get built because teams cannot discover what already exists. VS Code extensions are installed without vetting. AI agents and plugins pull data into external services without oversight. MCP servers are configured ad hoc by individual developers.

Os ambientes corporativos modernos sofrem com uma forma de dispersão específica e composta. APIs proliferam sem inventário central. Shadow APIs nascem porque os times não conseguem descobrir o que já existe. Extensões VS Code são instaladas sem vetting. Agentes de IA e plugins puxam dados para serviços externos sem supervisão. Servidores MCP são configurados ad hoc por cada desenvolvedor.

Los entornos empresariales modernos sufren una forma específica y compuesta de dispersión. Las APIs proliferan sin inventario central. Las shadow APIs nacen porque los equipos no pueden descubrir lo que ya existe. Las extensiones VS Code se instalan sin revisión. Los agentes de IA y plugins envían datos a servicios externos sin supervisión. Los servidores MCP se configuran ad hoc por cada desarrollador.

  • Security risk. Unvetted extensions and MCP servers may exfiltrate code or credentials.
  • Compliance risk. Sensitive data reaches AI model APIs without authorization.
  • Productivity risk. Developers duplicate APIs, use deprecated interfaces, and burn time on tooling setup.

2. The thesisA teseLa tesis

Nothing reaches a developer workstation without passing through a governed catalog. Azure API Center governs every API. The VS Code Private Marketplace governs every extension. Copilot governance covers every MCP server, agent, and prompt file. Three hubs, one control plane.
Nada chega à workstation do desenvolvedor sem passar por um catálogo governado. Azure API Center governa cada API. VS Code Private Marketplace governa cada extensão. Governança Copilot cobre cada MCP, agente e prompt. Três hubs, um plano de controle.
Nada llega a la estación de trabajo sin pasar por un catálogo gobernado. Azure API Center gobierna cada API. El VS Code Private Marketplace gobierna cada extensión. La gobernanza Copilot cubre cada MCP, agente y prompt. Tres hubs, un plano de control.

The architecture covers regulated industries, air-gapped scenarios, and the full integrated path from API design to consumption inside the IDE.

3. The shape of the runbookA forma do runbookLa forma del runbook

  • 3 governance hubs. Azure API Center, VS Code Private Marketplace, Copilot governance.
  • 12 sections. From sprawl analysis to operational runbook and governance framework.
  • 6 personas. Platform engineer, IT admin, security engineer, dev enablement lead, enterprise architect, API program manager.
  • 3 deployment scenarios. Full integrated, air-gapped, regulated industries (finance and public sector).
  • Day 0, Day 1, Day 2. Planning checklist, deployment sequence, developer onboarding, ongoing operations.

The runbook is approved at version 1.0.0 and structured for read-by-persona. Most teams start at section 2 (architecture) and section 11 (operational runbook).
