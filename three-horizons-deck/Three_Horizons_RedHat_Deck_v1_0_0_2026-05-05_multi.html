The five-layer pyramid.

Each layer depends on the one below it and is only meaningful when that dependency is satisfied. The stacking rule is not a metaphor; it is observable in every failed pilot.

You cannot build layer N+1 faster than layer N. You can only build them in parallel if you accept the debt that comes from the gap.

Layer 1 - foundation

Platform Engineering. The substrate.

The self-service, policy-governed, observable substrate that abstracts infrastructure complexity into developer-usable primitives. Its outputs are Golden Paths, Guardrails, Safety Nets, and Manual Review Workflows.

L1 deliverables
Red Hat Developer Hub portal - CI/CD as code - Policy-as-Code (Red Hat Advanced Cluster Security + Azure Policy) - Observability (Prometheus, Grafana, OpenTelemetry) - GitOps (OpenShift GitOps) - Workload Identity - Cost attribution - Service catalog - Documentation-as-code
Layer 2 - context

Context Layer. Codified knowledge.

The codified representation of the enterprise's knowledge, data, and decisions in machine-consumable form. Depends on the platform layer because the platform owns the primitives (RBAC, observability, lifecycle) that make context trustworthy.

L2 deliverables
RHDH service catalog - TechDocs - MCP servers (Azure, GitHub, observability) - Vector store (pgvector, Azure AI Search) - Lineage tracking (Purview, DataHub) - Embedding pipelines - Freshness metrics
Layer 3 - cognitive

Cognitive Layer. Models and reasoning.

The models and reasoning services that consume context to produce outputs. Foundation models (GPT-4o, Claude, Llama, Mistral), fine-tuned models, embedding services, and evaluation pipelines. Mediated through a model gateway that the platform layer provisions and governs.

L3 deliverables
Azure AI Foundry / GitHub Models gateway - Per-model cost and usage telemetry - Evaluation framework - Prompt logging - Output filtering - Cost circuit breakers - Model versioning
Layer 4 - intent

Intent Layer. Codified goals.

The codified representation of what the enterprise wants, expressed in a form that agents can plan against. Intent without context is hallucination; context without intent is description without direction.

L4 deliverables
Spec-Driven Development (Spec-Kit, Specky) - EARS notation requirements - Constitutional AI principles - Agent permission scope contracts - Issue templates encoding workflows - Microsoft Agent Governance Toolkit
Layer 5 - agentic

Agentic Layer. Autonomous execution.

Autonomous, goal-driven agents that combine intent, context, and cognition to take action. This is the visible layer for most AI investment in 2026, and it is the layer most likely to fail when the layers beneath it are missing.

L5 deliverables
GitHub Copilot Chat agents - foundry-agent template - multi-agent-system orchestration - sre-agent-integration - rag-application - Agent identity (Workload Identity, SPIFFE) - Agent observability - Agent promotion gates
The stacking rule

Layers stack in strict dependency order.

"You cannot build layer N+1 faster than layer N. You can only build them faster in parallel if you are willing to carry the debt that comes from the gap."

An organization that jumps to the agentic layer without the platform, context, and intent layers in place does not produce agents. It produces an amplified mess. Three Horizons collapses the time to build layer 1 so that layers 2-5 can be built earlier with less debt.

CNCF Four Pillars.

Golden Paths, Guardrails, Safety Nets, Manual Review. Originally designed for human-driven delivery. In 2026, re-expressed as the control plane for AI-driven delivery.

CNCF Four Pillars overview

The control taxonomy of platform engineering.

Pillar 01
Golden Paths
User interface
Opinionated, self-service workflows that get a developer from "I want to build X" to a working, compliant system in minutes.
Pillar 02
Guardrails
Safety interlocks
Mechanisms that prevent developers from deploying non-compliant or unsafe configurations. Preventive, not detective.
Pillar 03
Safety Nets
Immune system
Mechanisms that detect and recover from failures that guardrails did not prevent. Detective and corrective.
Pillar 04
Manual Review
Accountability
Explicit human-in-the-loop checkpoints for decisions that should not be automated. More important when AI accelerates everything.
Pillar 01 - Golden Paths

Templates that compress decisions.

A Golden Path is not a template; it is a template that encodes the enterprise's current best practice for a specific outcome, versioned and improvable. Single developer-facing entry point, parametrization for legitimate variability, embedded compliance, explicit deprecation.

For agents
Golden Paths become the templates by which agents are instantiated. An agent is a service; it deserves a scaffolder template. If it is not a Golden Path, the agent does not run it.
Pillar 02 - Guardrails

Make the wrong thing hard to do.

Policies enforced at admission. Policy-as-Code (Red Hat Advanced Cluster Security with built-in admission controllers and Azure Policy), Network Security Groups, Workload Identity (replacing long-lived service credentials), image and supply-chain policy with admission controllers rejecting unsigned workloads.

For agents
Guardrails become the policies agents must satisfy at runtime: permission scope, data access, output filtering. The agent containment field.
Pillar 03 - Safety Nets

Detect, recover, contain blast radius.

GitOps reconciliation (OpenShift GitOps with Argo CD) that returns cluster state to declared state. Observability stacks (Prometheus, Grafana, OpenTelemetry) that surface anomalies. Backup and recovery (Velero). Progressive delivery (canary, blue-green, feature flags).

For agents
Safety Nets become the reconciliation loops that correct agent drift: prompt-injection recovery, cost circuit breakers, output validation, SLO-driven freeze.
Pillar 04 - Manual Review

Human-in-the-loop on irreversible actions.

Environment promotion gates (dev auto-sync, staging gated, production requires approval). Change Advisory Board flows for high-impact changes. Cost gates above threshold. AI agent promotion gates that require review before scope or permission expansion.

For agents
Manual Review becomes the approval workflow for agent capability expansion. The AI equivalent of an elevated production deploy. Accountability stays with named humans by design.

Part IV
IV

Failure modes.

When the platform layer is weak, specific repeatable failure modes appear. The same structural cause: AI operates faster than ad-hoc processes can govern.

Failure 01 - triple debt

AI accumulates three forms of debt simultaneously.

01
Technical debt
Code and infrastructure shortcuts that raise future maintenance cost. AI-generated code without code review discipline.
02
Cognitive debt
Knowledge that exists only in a person's head, un-codified and unreviewed. Context fragmented across prompts and Slack threads.
03
Intent debt
Goals and policies that are implicit, contradictory, or undocumented. Agents act without contracts, outcomes drift from goals.
Failure 02 - shadow platforms

When the central platform is unusable, teams build their own.

Symptom

Each team reinvents the foundation

  • Bespoke CI/CD per team, each maintained by one person
  • Conflicting Terraform modules, no shared module registry
  • Five different MCP server forks, no canonical registry
  • Each squad pays for its own observability stack
Cost

2-3x the budget of a real platform

  • Distributed cost is invisible in any single budget line
  • Aggregate engineering hours dwarf a centralized team
  • Vendor sprawl, duplicated licenses, no negotiation leverage
  • First incident exposes the full bill
Failure 03 - context rot

Enterprise context silently becomes stale and wrong.

Service catalogs, documentation, lineage all rot ambiently in every enterprise. What platform engineering adds is the tooling to measure rot (freshness metrics, ownership coverage, documentation coverage) and the workflow to correct it. Without that tooling, AI agents reason over rotted context and produce plausible but wrong answers at scale.

Failure 04 - security regression

AI before platform security: +38% exploitable vulnerabilities.

CrowdStrike's 2026 Global Threat Report documents a specific regression: organizations that deployed AI coding assistants before maturing their platform security posture experienced a 38% increase in exploitable vulnerabilities in the first 12 months.

Resolution
The regression disappears in organizations where the platform enforces supply-chain and policy controls at admission. Signed images, secret scanning, dependency checks, IaC linting at PR time, not after deploy.
Failure 05 - the 100:1 problem

By 2028, agent-to-human ratio reaches 100:1.

If each agent requires a human to manage its lifecycle, credentials, and audit, the model does not scale. The only structural answer is a platform that treats agents as first-class workloads and governs them at platform speed rather than human speed.

Structural cause
Every failure mode has the same structural cause: the rate at which AI operates has exceeded the rate at which ad-hoc, non-platform processes can govern it. The answer is not to slow AI. The answer is to accelerate the platform.
Part V
V

The 2026 Dual Mandate.

Platform teams in 2026 are simultaneously responsible for two workstreams: augmenting the platform with AI (internal velocity) and enabling AI workloads at scale (external velocity).

Mandate A

Augment the platform with AI.

Use AI to make the platform itself better. Agents that triage alerts, draft runbook fixes, propose Terraform changes, route incidents. In-IDE agents that generate Golden Path invocations, explain platform errors, produce boilerplate.

Goal
Raise the internal velocity of the platform team so platform improvements ship faster. Three Horizons ships 17 GitHub Copilot Chat agents (plus Red Hat Lightspeed for OpenShift operators) pre-wired for Mandate A.
Mandate B

Enable AI workloads at scale.

Expose AI capabilities (inference, vector storage, agent runtime, evaluation, observability) as first-class platform primitives that application teams consume through Golden Paths.

Goal
Raise the external velocity of application teams so AI-using products ship faster. A platform that executes only Mandate A produces a faster platform team. A platform that executes only Mandate B becomes the bottleneck. Mature platforms execute both.
