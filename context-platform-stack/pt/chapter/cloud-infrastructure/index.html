01 Camada 01

Fundação de cloud e infraestrutura

Em que agentes rodam? O substrato de compute, rede e segurança que torna workloads agênticos observáveis e governáveis.

Diagnostic question
Em que agentes rodam?
18 min de leitura 9 seções

A premissa

Agentes não são microserviços. As premissas de infraestrutura que funcionaram na última década de workloads cloud-native quebram de maneiras específicas e previsíveis quando o workload é um loop de decisão guiado por LLM com invocações de tool não-determinísticas.

A primeira camada da pilha é o substrato de compute, rede, observabilidade e segurança que precisa existir antes que qualquer coisa acima possa ser racionalizada. Times que pulam essa camada terminam rodando agentes em laptops de dev, atrás de túneis ngrok, sem trilha de auditoria. Funciona por duas semanas. Aí o tráfego de produção chega.

O que muda com workloads agênticos

A tabela abaixo é o framing mais útil que tenho para explicar a times de infra por que “simplesmente coloca no nosso Kubernetes existente” é resposta incompleta.

A topologia do pod

O deployment canônico de agente em Kubernetes não é um container único. É um pod de quatro containers em que cada container mapeia para uma preocupação e pode ser operado, escalado e versionado de forma independente.

O container agent-runtime possui o client LLM, o scratchpad e o loop de decisão. Ele não fala com APIs de tool diretamente. Em vez disso, fala com o mcp-sidecar, um gateway MCP local que despacha para tool servers upstream, aplica RBAC e grava toda invocação no audit log. O container context-store expõe uma memória de três níveis (hot, warm, cold), descrita no Capítulo 3. O telemetry-shipper faz coleta OpenTelemetry com spans específicos de agente que cobrem chamada de modelo, chamada de tool e ponto de decisão.

A fronteira é deliberada. Se seu time de segurança bloqueia o LLM, você troca o runtime sem tocar em contexto ou tools. Se plataforma quer adicionar um novo guardrail, atualiza o sidecar. Se compliance pede retenção de trajetórias, você ajusta o telemetry shipper. Quatro preocupações, quatro deploys.

Agendar agentes não é agendar pods

O scheduling default do Kubernetes assume que um pod é uma unidade de trabalho. Para agentes, um pod é uma unidade de capacidade. Dois pods do mesmo agente rodando em paralelo podem coordenar numa tarefa compartilhada, ou conflitar num handle de tool compartilhado. O scheduler precisa saber.

Insight
Scheduling workload-aware é o novo gradiente

Plataformas agênticas iniciais estão convergindo num hint adicional de scheduling: um workload descriptor que declara a superfície esperada de tools, o budget máximo de tokens por invocação e o grupo de coordenação. O scheduler usa isso para fazer bin-packing de agentes por afinidade, não só por resource requests. Espere isso virar uma anotação padrão em 18 meses.

Anatomia de custo, o número que você provavelmente não mede

A maioria dos times de infra rastreia compute. Compute é 42% do custo de rodar um agente. Os outros 58% se escondem em lugares que não aparecem na conta do Kubernetes: inferência de tokens, I/O de armazenamento de contexto, chamadas de API de tool e overhead de observabilidade por agente.

Um time que só mede compute está voando cego em 58% do custo. Quando compute dobra, o time nota imediatamente. Quando o gasto com tool APIs triplica porque o agente começou a entrar em loop num endpoint que parecia barato, o time descobre pela área financeira no fechamento do trimestre.

O agente não inventa o formato do pod. Ele consome um blueprint. O blueprint é definido na Camada 02 (platform engineering). A requisição do agente é restringida pela Camada 04 (intent, expressa como CONSTITUTION.md). Se a camada de infraestrutura não expõe blueprints como CRDs first-class, todo agente reinventa o formato do pod, e metade deles estará errado.

Observabilidade tem que ser trajetória-primeiro

As semantic conventions OTel para GenAI foram lançadas no fim de 2025 e agora são a baseline. Qualquer infraestrutura de agente que não emite spans gen_ai.* para cada chamada de modelo e tool.* para cada invocação não está pronta para produção. A query que você precisa responder às 2 da manhã é:

Trajetórias não são úteis só para debug. São seu artefato de compliance. Quando o regulador perguntar por que o agente aprovou um determinado pull request, a trajetória é a resposta. Se você não consegue produzi-la, não tem um sistema auditável.

Segunda-feira de manhã

Cinco coisas que você pode fazer esta semana sem orçamento novo:

  1. Nomeie os quatro containers, mesmo que três deles sejam TODO. Documente a fronteira entre runtime, MCP sidecar, context store e telemetry shipper num README, e marque como CPS-L1.
  2. Adicione os três spans. Instrumente qualquer agente existente com gen_ai.request.model, gen_ai.usage.*_tokens e tool.name. É a trajetória mínima viável.
  3. Compute o TCO real. Puxe a conta do último mês de compute, inferência de tokens, vector store, tool APIs e observabilidade. Some tudo. Compare com o que você reportou como “custo do agente”. O gap é o que essa camada conserta.
  4. Escreva o schema do workload descriptor. Mesmo que como spec em markdown, ainda não como CRD. O exercício de descrever toolSurface, maxTokensPerCall e coordinationGroup força o time a concordar no vocabulário antes do trabalho de plataforma começar.
  5. Escolha uma query que você não consegue responder hoje. Para a maioria dos times é “o que esse agente fez durante o incidente X”. Escreva a query que você gostaria de conseguir rodar. Ela vira o critério de aceite para seu investimento em observabilidade.
