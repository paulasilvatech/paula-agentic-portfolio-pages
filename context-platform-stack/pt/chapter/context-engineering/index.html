Context engineering

O que agentes podem saber? O padrão ACE, memória de três níveis, skills como capacidades tipadas, e MCP como o protocolo que amarra tudo.

Diagnostic question
O que agentes podem saber?
20 min de leitura 9 seções

A premissa

Context engineering não é prompt engineering com mais passos. Prompt engineering otimiza uma única chamada LLM. Context engineering otimiza a ecologia de informação na qual um agente vive através de muitas chamadas durante muitos dias.

A disciplina tem três partes: o que o agente pode recuperar (RAG mais fontes estruturadas), o que o agente pode lembrar (memória de três níveis) e o que o agente pode fazer (skills). O protocolo que conecta os três é MCP. Times que tratam qualquer um desses como adendo terminam com agentes tecnicamente impressionantes em demos e praticamente inúteis em produção.

O padrão ACE

O padrão dominante para contexto em produção é ACE (Augmented Context Engineering). Tem três componentes: um retriever (tipicamente RAG sobre índice vetorial), um ranker (geralmente um cross-encoder pequeno que re-pontua os top-k candidatos) e um compressor (uma chamada LLM que sumariza para caber na context window).

O resultado de benchmark que fez o ACE pegar: 108 mil linhas de código C# em repositório enterprise, 283 sessões de agente de code review, 32% de redução de tokens comparado à abordagem ingênua de “enfiar tudo no prompt”, com scores de correctness equivalentes ou melhores. A lição é que curar contexto bate expandir contexto.

Memória de três níveis

Agentes precisam de três stores de memória diferentes com três políticas de atualização diferentes. Misturá-las é um erro comum que deixa agentes simultaneamente esquecidos e defasados.

Insight
Enforcement de budget não é opcional

Agentes em produção sem budgets explícitos de token por tier vão queimar a context window dentro de uma semana. O budget tem que ser aplicado pelo container de context store, não pelo template de prompt. Enforcement no nível de prompt falha na primeira vez que o dev esquece de incluir o budget numa nova skill.

Skills como capacidades tipadas

Uma skill é uma unidade nomeada, versionada e testável de capacidade do agente. Pense nela como uma assinatura de função para o LLM. O agente não chama um tool diretamente. Declara uma skill como dependência, e a plataforma resolve para um servidor MCP, uma política RBAC, uma quota e um suite de testes.

MCP como protocolo de fio

MCP (Model Context Protocol) é o formato de fio que conecta agentes às suas tools. Antes do MCP, todo framework de agente reinventava o formato de tool call. Agora existe um único protocolo com um único registry e uma única superfície de auditoria.

O MCP sidecar no pod do agente faz três coisas:

  1. Descobre servidores MCP registrados via o registry da plataforma
  2. Aplica RBAC e quota em toda invocação
  3. Loga toda chamada como span estruturado para replay de trajetória

Se você tem que escolher uma coisa para investir nesta camada, é MCP. Tudo mais (tiers de memória, skills, retrievers) pode ser específico por tool. MCP é o substrato que torna tudo componível.

AGENTS.md e o gap de curadoria humana

Um achado contraintuitivo de 2026: arquivos AGENTS.md gerados por LLM (o guia legível para agentes trabalhando num repo) performam 3% pior que sessões sem AGENTS.md nenhum. Arquivos AGENTS.md curados por humanos produzem 28,64% de redução de runtime e 16,58% de redução de tokens.

Watch out
Orce 2 a 3 horas por semana por portfolio de 10 agentes para curadoria humana

Curar AGENTS.md não é automatizável. É a camada de documentação que agentes leem antes de planejar. A intuição é a mesma que escrever boa doc de onboarding para um novo engenheiro. Se você terceiriza para um LLM, recebe conselho genérico que o agente teria inferido sozinho, mais uma falsa sensação de especificidade.

Como o Copilot usa skills em runtime

O agente não decidiu o que checar. A skill decidiu. A skill estava versionada, testada e ligada a uma constituição. Se amanhã o time mudar seus padrões de código, atualizam a definição da skill, sobem a versão e todo agente que usa herda a mudança sem uma única edição de prompt.

Segunda-feira de manhã

  1. Nomeie seus três tiers de memória. Mesmo que hoje sejam uma única tabela Postgres. Atribua a cada um um budget e um TTL. Só isso já pega metade dos bugs.
  2. Escreva um spec de skill. Escolha a capacidade mais invocada que você tem. Escreva o YAML. Não precisa implementar o registry ainda; o spec é a forcing function.
  3. Audite AGENTS.md. Se tem um, cheque se foi escrito por humano. Se não, reescreva ou delete. Doc falsa é pior que nenhuma doc.
  4. Registre um servidor MCP. Mesmo que você tenha que escrever um wrapper numa API REST existente. O exercício de versionar e escopar torna o registry concreto.
  5. Meça custo de contexto. Rode um agente em produção com log de trajetória por uma semana. Calcule o custo em dólar de retrieval de contexto vs inferência de modelo. Você vai se surpreender com qual domina.
