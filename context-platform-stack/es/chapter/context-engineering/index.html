03 Capa 03

Context engineering

¿Qué pueden saber los agentes? El patrón ACE, memoria de tres niveles, skills como capacidades tipadas, y MCP como el protocolo que conecta todo.

Diagnostic question
¿Qué pueden saber los agentes?
20 min de lectura 9 secciones

La premisa

Context engineering no es prompt engineering con más pasos. Prompt engineering optimiza una sola llamada LLM. Context engineering optimiza la ecología de información en la que un agente vive a través de muchas llamadas durante muchos días.

La disciplina tiene tres partes: lo que el agente puede recuperar (RAG más fuentes estructuradas), lo que el agente puede recordar (memoria de tres niveles) y lo que el agente puede hacer (skills). El protocolo que conecta los tres es MCP. Los equipos que tratan cualquiera de estos como después-pensado terminan con agentes técnicamente impresionantes en demos y prácticamente inútiles en producción.

El patrón ACE

El patrón dominante para contexto en producción es ACE (Augmented Context Engineering). Tiene tres componentes: un retriever (típicamente RAG sobre índice vectorial), un ranker (normalmente un cross-encoder chico que re-puntúa los top-k candidatos) y un compressor (una llamada LLM que resume para entrar en la context window).

El resultado de benchmark que hizo que ACE pegue: 108 mil líneas de C# en un repo enterprise, 283 sesiones de agente de code review, 32% de reducción de tokens comparado al enfoque ingenuo de “meter todo en el prompt”, con scores de correctness equivalentes o mejores. La lección es que curar contexto le gana a expandir contexto.

Memoria de tres niveles

Los agentes necesitan tres stores de memoria diferentes con tres políticas de actualización diferentes. Mezclarlas es un error común que deja a los agentes simultáneamente olvidadizos y desactualizados.

Insight
Enforcement de budget no es opcional

Los agentes en producción sin budgets explícitos de tokens por tier van a quemar la context window en una semana. El budget tiene que aplicarse desde el contenedor de context store, no desde el template de prompt. El enforcement a nivel de prompt falla la primera vez que el dev se olvida de incluir el budget en una nueva skill.

Skills como capacidades tipadas

Una skill es una unidad nombrada, versionada y testeable de capacidad del agente. Pensala como una firma de función para el LLM. El agente no llama a una tool directamente. Declara una skill como dependencia, y la plataforma resuelve a un servidor MCP, una política RBAC, una cuota y un suite de tests.

MCP como protocolo de cable

MCP (Model Context Protocol) es el formato de cable que conecta agentes a sus tools. Antes de MCP, cada framework de agente reinventaba la forma de tool call. Ahora hay un solo protocolo con un solo registry y una sola superficie de auditoría.

El MCP sidecar en el pod del agente hace tres cosas:

  1. Descubrir servidores MCP registrados vía el registry de plataforma
  2. Aplicar RBAC y cuota en cada invocación
  3. Loguear cada llamada como span estructurado para replay de trayectoria

Si tenés que elegir una cosa en la que invertir en esta capa, es MCP. Todo lo demás (memory tiers, skills, retrievers) puede ser específico por tool. MCP es el sustrato que hace que todo sea componible.

AGENTS.md y la brecha de curaduría humana

Un hallazgo contraintuitivo de 2026: los archivos AGENTS.md generados por LLM (la guía legible para agentes trabajando en un repo) performan 3% peor que sesiones sin AGENTS.md alguno. Los archivos AGENTS.md curados por humanos producen 28,64% de reducción de runtime y 16,58% de reducción de tokens.

Watch out
Presupuestá 2 a 3 horas por semana por portfolio de 10 agentes para curaduría humana

Curar AGENTS.md no es automatizable. Es la capa de documentación que los agentes leen antes de planear. La intuición es la misma que escribir buena doc de onboarding para un nuevo ingeniero. Si lo terceriás a un LLM, recibís consejo genérico que el agente habría inferido solo, más una falsa sensación de especificidad.

Cómo Copilot usa skills en runtime

El agente no decidió qué chequear. La skill decidió. La skill estaba versionada, testeada y atada a una constitución. Si mañana el equipo cambia sus estándares de código, actualizan la definición de skill, suben la versión, y cada agente que la usa hereda el cambio sin una sola edición de prompt.

Lunes por la mañana

  1. Nombrá tus tres tiers de memoria. Aunque hoy sean una sola tabla Postgres. Asigná a cada uno un budget y un TTL. Solo eso ya agarra la mitad de los bugs.
  2. Escribí un spec de skill. Elegí la capacidad más invocada que tengas. Escribí el YAML. No necesitás implementar el registry todavía; el spec es la forcing function.
  3. Auditá AGENTS.md. Si tenés uno, chequeá si lo escribió un humano. Si no, reescribilo o borralo. Doc falsa es peor que sin doc.
  4. Registrá un servidor MCP. Aunque tengas que escribir un wrapper sobre una API REST existente. El ejercicio de versionar y escopear hace al registry concreto.
  5. Medí costo de contexto. Corré un agente de producción con log de trayectoria por una semana. Calculá el costo en dólares de retrieval de contexto vs inferencia de modelo. Te vas a sorprender con cuál domina.
