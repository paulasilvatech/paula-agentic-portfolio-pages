01 Capa 01

Fundación de cloud e infraestructura

¿En qué corren los agentes? El sustrato de compute, red y seguridad que hace que los workloads agénticos sean observables y gobernables.

Diagnostic question
¿En qué corren los agentes?
18 min de lectura 9 secciones

La premisa

Los agentes no son microservicios. Las premisas de infraestructura que funcionaron en la última década de workloads cloud-native se rompen de maneras específicas y predecibles cuando el workload es un loop de decisión guiado por LLM con invocaciones de tool no determinísticas.

La primera capa del stack es el sustrato de compute, red, observabilidad y seguridad que debe existir antes de que cualquier cosa arriba pueda razonarse. Los equipos que saltan esta capa terminan corriendo agentes en laptops de dev, detrás de túneles ngrok, sin trail de auditoría. Funciona dos semanas. Después llega el tráfico de producción.

La topología del pod

El deployment canónico de agente en Kubernetes no es un contenedor único. Es un pod de cuatro contenedores donde cada uno mapea a una preocupación y puede operarse, escalarse y versionarse independientemente.

El contenedor agent-runtime tiene el cliente LLM, el scratchpad y el loop de decisión. No habla con APIs de tool directamente. Habla con el mcp-sidecar, un gateway MCP local que despacha a tool servers upstream, aplica RBAC y escribe cada invocación en el audit log. El context-store expone una memoria de tres niveles (hot, warm, cold), descrita en el Capítulo 3. El telemetry-shipper hace colección OpenTelemetry con spans específicos de agente que cubren llamada de modelo, llamada de tool y punto de decisión.

La frontera es deliberada. Si seguridad bloquea el LLM, reemplazás el runtime sin tocar contexto o tools. Si plataforma quiere agregar un nuevo guardrail, actualizan el sidecar. Si compliance pide retención de trayectorias, ajustás el telemetry shipper. Cuatro preocupaciones, cuatro deploys.

Schedular agentes no es schedular pods

El scheduling default de Kubernetes asume que un pod es una unidad de trabajo. Para agentes, un pod es una unidad de capacidad. Dos pods del mismo agente corriendo en paralelo pueden coordinar en una tarea compartida, o chocar en un handle de tool compartido. El scheduler necesita saberlo.

Insight
Scheduling workload-aware es el nuevo gradiente

Las plataformas agénticas iniciales están convergiendo en un hint adicional de scheduling: un workload descriptor que declara la superficie esperada de tools, el budget máximo de tokens por invocación y el grupo de coordinación. El scheduler lo usa para hacer bin-packing de agentes por afinidad, no solo por resource requests. Esperá que esto sea una anotación estándar en 18 meses.

Anatomía de costo, el número que probablemente no medís

La mayoría de los equipos de infra rastrea compute. Compute es el 42% del costo de correr un agente. El otro 58% se esconde en lugares que no aparecen en la cuenta de Kubernetes: inferencia de tokens, I/O de almacenamiento de contexto, llamadas de API de tool y overhead de observabilidad por agente.

APIs de tool y observabilidad

Un equipo que solo mide compute está volando a ciegas en el 58% del costo. Cuando compute se duplica, el equipo lo nota inmediatamente. Cuando el gasto en tool APIs se triplica porque el agente empezó a loopear en un endpoint que parecía barato, el equipo se entera por finanzas al cierre del trimestre.

El agente no inventa la forma del pod. Consume un blueprint. El blueprint está definido en la Capa 02 (platform engineering). Si la capa de infraestructura no expone blueprints como CRDs first-class, cada agente reinventa la forma del pod, y la mitad va a estar mal.

La observabilidad debe ser trayectoria-primero

Las trayectorias no solo sirven para debug. Son tu artefacto de compliance. Cuando el regulador pregunte por qué el agente aprobó un pull request, la trayectoria es la respuesta. Si no la podés producir, no tenés un sistema auditable.

Lunes por la mañana

Cinco cosas que podés hacer esta semana sin presupuesto nuevo:

  1. Nombrá los cuatro contenedores, aunque tres sean TODO. Documentá la frontera entre runtime, MCP sidecar, context store y telemetry shipper en un README, y taggeá como CPS-L1.
  2. Agregá los tres spans. Instrumentá cualquier agente existente con gen_ai.request.model, gen_ai.usage.*_tokens y tool.name. Es la trayectoria mínima viable.
  3. Calculá el TCO real. Tomá la cuenta del último mes de compute, inferencia de tokens, vector store, tool APIs y observabilidad. Sumá todo. Comparalo con lo que reportaste como “costo del agente”. La brecha es lo que esta capa arregla.
  4. Escribí el schema del workload descriptor. Aunque sea como spec en markdown, todavía no como CRD. El ejercicio de describir toolSurface, maxTokensPerCall y coordinationGroup fuerza al equipo a acordar vocabulario antes que empiece el trabajo de plataforma.
  5. Elegí una query que no podés responder hoy. Para la mayoría de los equipos es “qué hizo ese agente durante el incidente X”. Escribí la query que te gustaría correr. Se vuelve el criterio de aceptación para tu inversión en observabilidad.
