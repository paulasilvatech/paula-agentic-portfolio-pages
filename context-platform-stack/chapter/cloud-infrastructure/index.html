Cloud and infrastructure foundation

What can agents run on? The compute, networking, and security substrate that makes agentic workloads observable and governable.

Diagnostic question
What can agents run on?
18 min read 9 sections

The premise

Agents are not microservices. The infrastructure assumptions that worked for the last decade of cloud-native workloads break in specific, predictable ways when the workload is an LLM-driven decision loop with non-deterministic tool invocations.

The first layer of the stack is the compute, networking, observability, and security substrate that has to exist before anything above it can be reasoned about. Teams that skip this layer end up running agents on dev laptops, behind ngrok tunnels, with no audit trail. That works for two weeks. Then production traffic arrives.

What changes with agentic workloads

The pod topology

The canonical agent deployment on Kubernetes is not a single container. It is a four-container pod where each container maps to one concern and can be operated, scaled, and versioned independently.

The agent-runtime container owns the LLM client, the scratchpad, and the decision loop. It does not talk to tool APIs directly. Instead, it talks to the mcp-sidecar, a local MCP gateway that fans out to upstream tool servers, enforces RBAC, and writes every invocation to the audit log. The context-store container exposes a three-tier memory (hot, warm, cold) described in Chapter 3. The telemetry-shipper does OpenTelemetry collection with agent-specific spans that cover model call, tool call, and decision point.

The boundary is deliberate. If your security team flags the LLM, you replace the runtime without touching context or tools. If your platform team wants to add a new guardrail, they update the sidecar. If compliance asks for trajectory retention, you tune the telemetry shipper. Four concerns, four deploys.

Scheduling agents is not scheduling pods

Default Kubernetes scheduling assumes a pod is a unit of work. For agents, a pod is a unit of capability. Two pods of the same agent running in parallel can coordinate on a shared task, or they can conflict on a shared tool handle. The scheduler needs to know.

Insight
Workload-aware scheduling is the new gradient

Early agentic platforms are converging on an additional scheduling hint: a workload descriptor that declares the expected tool surface, the max-tokens budget per invocation, and the coordination group. The scheduler uses this to bin-pack agents by affinity, not just by resource requests. Expect this to become a standard annotation within 18 months.

Cost anatomy, the number you are probably not measuring

Most infra teams track compute. Compute is 42% of the cost of running an agent. The other 58% hides in places that do not show up on the Kubernetes bill: token inference, context storage I/O, tool API calls, and per-agent observability overhead.

28%

15%

A team that only measures compute is flying blind on 58% of the cost. When compute doubles, the team notices immediately. When tool-API spend triples because the agent started looping through a cheap-looking endpoint, the team finds out from finance at end of quarter.

Monday morning

Five things you can do this week with no new budget:

  1. Name the four containers, even if three of them are TODO. Document the boundary between runtime, MCP sidecar, context store, and telemetry shipper in a README, and tag it as CPS-L1.
  2. Add the three spans. Instrument any existing agent with gen_ai.request.model, gen_ai.usage.*_tokens, and tool.name. That is the minimum viable trajectory.
  3. Compute the real TCO. Pull last month’s bill for compute, token inference, vector store, tool APIs, and observability. Add them up. Compare with what you reported as “agent cost.” The gap is what this layer fixes.
  4. Write the workload descriptor schema. Even as a markdown spec, not a CRD yet. The exercise of describing toolSurface, maxTokensPerCall, and coordinationGroup forces your team to agree on vocabulary before platform work starts.
  5. Pick one query you cannot currently answer. For most teams it is “what did this agent do during incident X.” Write the query you wish you could run. That becomes the acceptance criterion for your observability investment.
