Intent engineering

What should agents optimize for? CONSTITUTION.md, spec-driven development in EARS, and intent debt as the metric nobody measures yet.

Diagnostic question
What should agents optimize for?
18 min read 5 sections

The premise

Intent is the top layer and the rarest. Most enterprises have infrastructure. A good number have platform. Some have context. Almost none have intent engineered as a first-class artifact. Intent is usually a Slack thread between a PM and an engineer, which the agent never sees.

This chapter makes the case that intent is code. Specifically, it is two artifacts: a constitution that describes the non-negotiable constraints, and specifications written in a structured, machine-testable format (EARS) that describe what the agent should accomplish. Everything else is commentary.

CONSTITUTION.md

A constitution is a plain markdown file that lives at the root of your agents repository. It describes, in natural language, the rules that every agent under its jurisdiction must honor. Platform guardrails enforce these rules automatically at runtime.

The constitution is versioned like code. It ships through pull requests. It is signed by named authorities. It is validated on every commit via automated checks that verify each clause has a corresponding policy-as-code enforcement artifact. If a clause exists in markdown but has no enforcement, the CI job fails.

Specification Driven Development in EARS

The second intent artifact is the specification. Every non-trivial agent task should be written as an EARS (Easy Approach to Requirements Syntax) spec before implementation. EARS gives you a handful of templates that are machine-testable.

Intent debt, the metric nobody measures yet

Intent debt is the gap between what you said the agent should do and what it is actually doing. It accumulates silently. An agent that learns to cut “harmless” shortcuts today will cut a catastrophic shortcut in three months.

Mature organizations measure ID weekly and treat it as defect backlog. The formula:

Insight
ID = Σ (D × S) across recent decisions

For every decision the agent makes, compute a drift score D (how far the decision falls from what the constitution implies, between 0 and 1) and a stake score S (the business impact of that decision, between 0 and 10). The weekly ID is the sum.

Monday morning

  1. Write CONSTITUTION.md. Three clauses is a legitimate start. Name the authorities. Ship as a pull request.
  2. Pick one spec you wish existed. Write it in EARS. You will find that 30% of your “requirements” turn out to be unclear.
  3. Compute one week’s intent debt. Manually if necessary. The number will shock you on the first measurement.
  4. Identify your top-stake decisions. Which agent decisions have the highest S? Those are the ones you audit first.
  5. Commit to the cadence. Intent debt measured monthly is vanity. Measured weekly is operational. Pick the cadence and stick to it.
