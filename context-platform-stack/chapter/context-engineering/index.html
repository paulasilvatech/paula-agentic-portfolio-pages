Context engineering

What can agents know? The ACE pattern, three-tier memory, skills as typed capabilities, and MCP as the protocol that ties them together.

Diagnostic question
What can agents know?
20 min read 9 sections

The premise

Context engineering is not prompt engineering with more steps. Prompt engineering optimizes a single LLM call. Context engineering optimizes the information ecology an agent lives inside across many calls over many days.

The discipline has three parts: what the agent can retrieve (RAG plus structured sources), what the agent can remember (three-tier memory), and what the agent can do (skills). The protocol that connects all three is MCP. Teams that treat any of these as afterthoughts end up with agents that are technically impressive in demos and practically useless in production.

The ACE pattern

The dominant pattern for production context is ACE, which stands for Augmented Context Engineering. It has three components: a retriever (typically RAG over a vector index), a ranker (usually a small cross-encoder that re-scores the top-k candidates), and a compressor (an LLM call that summarizes to fit the context window).

The benchmark result that made ACE stick: 108 thousand lines of C# code in an enterprise repository, 283 sessions of a code-review agent, 32% token reduction compared to naive “stuff everything in the prompt” approaches, with equivalent or better correctness scores. The lesson is that curating context beats expanding context.

Three-tier memory

Agents need three different memory stores with three different update policies. Conflating them is a common mistake that makes agents simultaneously forgetful and stale.

Insight
Budget enforcement is not optional

Production agents without explicit per-tier token budgets will burn through their context window inside a week. The budget has to be enforced by the context-store container, not by the prompt template. Prompt-level enforcement fails the first time the developer forgets to include the budget in a new skill.

Skills as typed capabilities

A skill is a named, versioned, testable unit of agent capability. Think of it as a function signature for the LLM. The agent does not call a tool directly. It declares a skill as a dependency, and the platform resolves the skill to an MCP server, an RBAC policy, a quota, and a test suite.

MCP as the wire protocol

MCP (Model Context Protocol) is the wire format that connects agents to their tools. Before MCP, every agent framework reinvented the tool-call shape. Now there is a single protocol with a single registry and a single audit surface.

The MCP sidecar in your agent pod does three things:

  1. Discover registered MCP servers via the platform registry
  2. Enforce RBAC and quota on every invocation
  3. Log every call as a structured span for trajectory replay

If you have to choose one thing to invest in at this layer, it is MCP. Everything else (memory tiers, skills, retrievers) can be tool-specific. MCP is the substrate that makes them composable.

AGENTS.md and the human-curation gap

A counterintuitive finding from 2026: LLM-generated AGENTS.md files (the human-readable guide for agents working in a repo) perform 3% worse than sessions with no AGENTS.md at all. Human-curated AGENTS.md files produce 28.64% runtime reduction and 16.58% token reduction.

Watch out
Budget 2 to 3 hours per week per 10-agent portfolio for human curation

Curating AGENTS.md is not automatable. It is the documentation layer that agents read before they plan. The intuition is the same as writing good onboarding docs for a new engineer. If you outsource it to an LLM, you get generic advice that the agent would have inferred anyway, plus a false sense of specificity.

How Copilot uses skills at runtime

The agent did not decide what to check. The skill did. The skill was versioned, tested, and bound to a constitution. If tomorrow the team changes its coding standards, they update the skill definition, bump the version, and every agent using it inherits the change without a single prompt edit.

Monday morning

  1. Name your three memory tiers. Even if today they are all one Postgres table. Assign each one a budget and a TTL. That alone catches half the bugs.
  2. Write one skill spec. Pick the most-invoked capability you have. Write the YAML. You do not need to implement the registry yet; the spec is the forcing function.
  3. Audit AGENTS.md. If you have one, check whether it was written by a human. If not, rewrite it or delete it. Fake docs are worse than no docs.
  4. Register one MCP server. Even if you have to write a wrapper around an existing REST API. The exercise of versioning and scoping makes the registry concrete.
  5. Measure context cost. Run one production agent with trajectory logging for a week. Compute the dollar cost of context retrieval vs model inference. You will be surprised which dominates.
