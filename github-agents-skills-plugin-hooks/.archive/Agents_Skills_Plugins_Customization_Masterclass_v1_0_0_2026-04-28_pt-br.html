Parte I
I

Fundamentos.

Um agent é um runtime. Uma skill é uma instrução empacotada. Um plugin é uma unidade de distribuição. Três camadas, três responsabilidades, um ecossistema.

Conceito, agent

Um agent é LLM mais instruções mais ferramentas.

Um runtime autônomo que recebe um objetivo, decide qual ferramenta chamar, executa, observa o resultado e decide o próximo passo. Sem instruções, ele não tem comportamento. Sem ferramentas, ele não afeta o mundo. Sem LLM, ele não consegue raciocinar sobre nenhum dos dois.

01
LLM core
O motor de raciocínio. Lê o objetivo, escolhe ferramentas, interpreta saídas, decide o próximo passo.
02
Instruções
O contrato de comportamento. Define persona, escopo, tom, restrições e política de recusa.
03
Ferramentas
Lê arquivos, roda shell, chama APIs, consulta servidores MCP. As mãos do agent.
Hierarquia

Três camadas. Cada uma compõe a anterior.

PLUGINUnidade de distribuição. Pacote versionado que reúne agents, skills, hooks e servidores MCP sob um único manifesto. Usado para entregar uma configuração testada para o time inteiro de uma vez.
AGENTRuntime. A entidade autônoma que mantém a conversa, escolhe ferramentas e produz saída. Pode carregar zero, uma ou várias skills conforme a tarefa.
SKILLInstrução empacotada. Um arquivo SKILL.md mais arquivos de referência opcionais. Carregada sob demanda para manter a janela de contexto enxuta.
Motivação

Por que customizar? Porque resposta genérica não é a sua resposta.

Um LLM puro não conhece sua base de código, suas convenções, sua política de segurança ou a linguagem do seu domínio. Customização fecha essa lacuna. O mesmo prompt que precisa de dez mensagens em um chat genérico chega em uma com um agent ou skill bem escopado.

Sem customização

Resposta genérica, contexto repetido, drift

  • Cada prompt restabelece o mesmo contexto manualmente
  • Saídas perdem convenções e padrões da casa
  • Conhecimento fica nas pessoas, não no time
  • Sem reuso, sem versionamento, sem governança
Com customização

Contextualizado, reutilizável, governado

  • Contexto carregado uma vez, reusado pelo time
  • Saídas seguem seus padrões por padrão
  • Skills viajam com a base de código, versionadas em git
  • Revisáveis, auditáveis, melhoráveis ao longo do tempo
As três opções

O Copilot oferece três caminhos de customização.

Cada caminho tem custo, escopo e governança diferentes. Escolher o certo é a decisão mais importante na sua estratégia de IA. A escolha errada se manifesta como agent sprawl, skills abandonadas ou um generalista no qual ninguém confia.

Opção A
Custom agent + skill
Alta especialização. Reservado para domínios estreitos, persona, escopo e ferramental travados.
Opção B
Generalista + skill
O padrão para a maioria. Generalista carrega skills sob demanda. Reuso e flexibilidade maximizados.
Opção C
Generalista sozinho
Tarefas pontuais, exploração, scripts únicos. Não precisa de skill quando o contexto já é suficiente.
Parte II
II

Framework de decisão.

Antes de construir qualquer coisa, decida qual das três opções o problema realmente pede. A maioria vai para generalista mais skill. Custom agents são exceção, não regra.

Quando criar um custom agent

Crie um custom agent só quando pelo menos três disto forem verdade.

01A persona precisa ficar travada. O agent deve recusar tarefas fora do escopo, não negociá-las.
02Domínio sensível. Segurança, finanças, saúde, compliance. Resposta errada tem custo assimétrico.
03Superfície de ferramentas estreita. Apenas um conjunto fixo deve ser alcançável, independente do que o usuário pedir.
04Várias skills compõem um workflow. O agent orquestra na ordem conhecida, não deixa o LLM improvisar a sequência.
05Reusado por muita gente, muitas vezes. O investimento se amortiza em centenas de invocações por mês.
Matriz de decisão

Escolha a opção que o seu problema realmente tem.

Regra de ouro

Na dúvida, comece com uma skill.

Uma skill é barata de escrever, fácil de apagar e trivial de compartilhar. Um custom agent é um compromisso. Muitos times pulam direto para custom agent quando uma skill bem escopada no generalista resolveria em uma tarde. Promova uma skill a custom agent só depois que ela tiver provado seu valor e você tiver evidência de que a persona realmente precisa ficar travada.

Heurística

Se uma skill no generalista leva o time a 80% do caminho, entregue a skill. Os 20% restantes raramente justificam um custom agent.

Três times reais

Mesmo ferramental. Três escolhas diferentes.

Carlos, generalista

Maria, plataforma

Parte III
III

Custom agents e generalistas.

Um custom agent é frontmatter mais instruções. Um generalista é o mesmo agent que você já tem, só esperando o contexto certo. A maioria dos times subestima o segundo e superinveste no primeiro.

Anatomia do agent

Um arquivo de agent. Frontmatter mais corpo.

.github/agents/security-reviewer.md
---
name: security-reviewer
description: Revisa pull requests para vulnerabilidades do OWASP Top 10.
model: claude-sonnet-4-6
tools: [read_file, grep, list_directory]
applyTo: "**/*.py", "**/*.ts"
---

# Security Reviewer

Você é um revisor de código focado em segurança. Seu escopo é estrito:
revise pull requests procurando vulnerabilidades mapeadas ao OWASP Top 10.

## Política de recusa
Se pedirem para escrever feature, refatorar por estilo ou qualquer coisa
fora de segurança, recuse com cordialidade e direcione para o generalista.

## Formato de saída
Para cada finding: severidade, CWE ID, arquivo:linha, explicação, sugestão de fix.
Referência de frontmatter

Os campos que moldam o comportamento de fato.

Campo Tipo O que controla
namestringIdentificador único. Usado no seletor do chat e na lógica de roteamento.
descriptionstringPropósito em uma frase. Lido pelo meta-agent que decide quando sugerir este agent.
modelstringEm qual modelo o agent roda. Trava o roteamento para personas sensíveis.
toolsarrayFerramentas permitidas. Qualquer coisa fora da lista é inalcançável.
applyToglobPadrões de arquivo que ativam o agent automaticamente no chat.
versionstringVersão semântica. Subida em qualquer edição que muda comportamento.
O generalista, forças e limites

Capaz por padrão. Cego para suas especificidades.

Forças

Amplitude e iteração rápida

  • Lida com perguntas abertas e exploração
  • Adapta tom e profundidade no meio da conversa
  • Sem investimento prévio, disponível imediatamente
  • Fácil combinar várias skills num único chat
Limites

Genérico, sem guard rails

  • Não conhece suas convenções nem seu stack
  • Tom e escopo derrapam sem orientação explícita
  • Não dá para impor recusa de forma confiável
  • O conhecimento fica no chat, nunca é reusado
Arquivos .prompt.md

Workflows reutilizáveis sem sair do generalista.

Entre o prompt ad hoc e a skill completa fica o prompt file. Um .prompt.md é um template de uma única execução que o generalista aplica sob demanda. Use para workflows recorrentes que ainda não estão estáveis o bastante para virar skill, ou para tarefas que rodam de forma agendada.

.github/prompts/release-notes.prompt.md
---
name: release-notes
description: Gera release notes a partir de um git log entre duas tags.
---

Você está escrevendo release notes para o changelog público.
Agrupe commits em três seções: Features, Fixes, Internal.
Pule merge commits. Use voz ativa. Uma linha por item, sem marketing.

Input: {{git_log}}
Saída: Markdown com três seções H3, na ordem acima.
Parte IV
IV

Skills.

Uma skill é o componente reutilizável mais valioso do stack. Um arquivo SKILL.md mais material de referência opcional. Aprender a escrever skills bem é o investimento de maior alavancagem em todo o ecossistema.

O que é uma skill

Pense em skill como uma receita num livro de cozinha.

O livro fica na prateleira, indexado pela descrição. O agent só abre quando a tarefa em mãos casa com uma receita. Aberta, a receita é seguida passo a passo. Tarefa concluída, o livro volta para a prateleira. A memória de trabalho do agent fica limpa para a próxima tarefa.

01
Descobrível
Vive em diretório conhecido, indexada por descrição, encontrada pelo agent automaticamente.
02
Carga preguiçosa
Só a descrição fica em contexto até o agent decidir abrir. Corpo e referências esperam.
03
Reutilizável
Versionada em git, compartilhada no time, componível com outras skills.
Anatomia do SKILL.md

Frontmatter mais corpo. Só isso.

skills/code-reviewer/SKILL.md
---
name: code-reviewer
description: >
  Use esta skill sempre que o usuário pedir code review, busca por bugs,
  ou checagem de pull request. Triggers nas palavras "review", "PR",
  "code quality", ou quando um diff é colado no chat.
---

# Code Reviewer Skill

## Workflow
1. Leia o diff ou o arquivo sob revisão.
2. Verifique bugs, problemas de segurança, violações de PEP 8.
3. Agrupe findings por severidade: blocker, major, minor.
4. Sugira um fix concreto para cada blocker.

## Template de saída
Veja references/output-template.md para a estrutura exata.
Padrões de trigger

A descrição é como a skill ganha sua carga.

Descrição vaga é a razão mais comum de uma skill ficar sem uso. A descrição é a única coisa que o meta-agent vê quando decide se carrega esta skill. Seja específico. Liste palavras-gatilho. Nomeie tipos de arquivo. Diga em quais contextos se aplica e em quais não.

Descrição fraca

Carregada de forma inconsistente, ou nunca

  • "Ajuda com código"
  • "Revisa coisas"
  • "Para usuários de Python"
Descrição forte

Carregada com confiança, recusada com clareza

  • "Use ao revisar pull requests para o OWASP Top 10"
  • "Triggers nas palavras: review, PR, audit, security"
  • "Aplica em arquivos .py e .ts. Pula .md e configs."
Arquivos de referência

Mais poder, sem inflar o contexto.

SKILL.md deve ficar curto, idealmente abaixo de 200 linhas. Tudo maior vai para arquivos de referência. O corpo do SKILL.md diz ao agent qual referência abrir e quando. Referências são carregadas só sob demanda, pelo nome, quando o workflow precisa.

skills/code-reviewer/, estrutura
skills/code-reviewer/
├── SKILL.md                    # 100 linhas, sempre carregada quando triggered
├── references/
│   ├── owasp-top-10.md         # Carregada só em revisões de segurança
│   ├── style-guide.md          # Carregada só em revisões de estilo
│   └── output-template.md      # Carregada antes de produzir saída
└── examples/
    ├── good-finding.md
    └── bad-finding.md
Ciclo de vida

Do arquivo no disco ao resultado na tela, em quatro fases.

01Discovery. O runtime varre diretórios conhecidos, parseia o frontmatter de cada SKILL.md e monta um índice de nome mais descrição. Acontece uma vez no boot e a cada mudança em skills/.
02Loading. Quando o pedido do usuário casa com uma descrição, o meta-agent carrega o corpo do SKILL.md no contexto. Referências ficam fechadas.
03Execution. O agent segue o workflow, abre arquivos de referência conforme necessário, chama ferramentas, produz saída. Hooks de pré e pós-tool podem interceptar.
04Evolution. Feedback retorna. A skill é versionada, a descrição é refinada, exemplos são adicionados, ramos mortos são podados. Sem essa fase, a biblioteca apodrece.
Otimização de tokens

Carga preguiçosa não é opcional em escala.

Uma organização com 200 skills não pode bancar carregar todas em cada chat. O índice de discovery mantém só a descrição na memória de trabalho. O corpo abre sob demanda. Referências abrem ainda depois. Bem feito, uma biblioteca de 200 skills custa o mesmo orçamento de contexto que uma de 5, até o momento em que uma skill é de fato necessária.

Sempre carregado
~200
tokens por skill, só a descrição. Custo fixo do índice.
Carregado no match
~2k
tokens, o corpo do SKILL.md. Pago só quando a skill é selecionada.
Carregado sob demanda
~5k+
tokens por arquivo de referência, aberto pelo nome quando o workflow precisa.
Falhas comuns

Quando skills falham, geralmente é uma destas cinco.

Falha Sintoma Correção
Descrição vagaSkill nunca carrega, ou carrega quando não cabeAdicione palavras-gatilho, tipos de arquivo, fronteiras de escopo
Corpo longo demaisContexto enche, agent perde o fioMova detalhe para arquivos de referência, linke do corpo
Sem template de saídaFormato derrapa entre execuçõesAdicione seção explícita de saída ou arquivo de referência dedicado
Ferramenta não permitidaSkill diz "rode os testes" mas o agent não consegueCheque a lista de tools do agent pai, adicione o que falta
Skill nunca é usadaAdoção empaca, fica no repo sem usoAudite a descrição, adicione exemplos, escreva um PR-tutorial
Parte V
V

Plugins, MCP e hooks.

Plugins empacotam várias skills, agents e hooks para distribuição. MCP deixa o agent falar com o mundo. Hooks deixam você injetar comportamento em momentos do ciclo de vida.

Manifesto do plugin

plugin.json. A fonte única de verdade.

Um plugin é unidade de distribuição, não runtime. Empacota um conjunto opinado de agents, skills, hooks e servidores MCP sob um manifesto único. É a forma de entregar uma configuração testada para o time, instalar em um comando e versionar como unidade.

plugin.json
{
  "name": "code-excellence",
  "version": "1.2.0",
  "description": "Agents e skills para qualidade de código backend.",
  "author": "platform-team@empresa.com",
  "agents": ["./agents/security-reviewer.md"],
  "skills": ["./skills/code-reviewer", "./skills/api-tester"],
  "hooks": {
    "preToolUse": "./hooks/pre-tool-use.js",
    "postToolUse": "./hooks/post-tool-use.js"
  },
  "mcpServers": [
    { "name": "jira", "url": "https://jira.empresa.com/mcp" }
  ]
}
MCP e hooks

Como o agent fala com o mundo, e como você intercepta.

MCP, Model Context Protocol

Padrão para chamar o mundo externo

  • O agent fala um protocolo, qualquer servidor compatível fica endereçável
  • Tool list, tool call, tool result, sobre stdio ou HTTP
  • Embrulha sistemas reais: banco, ticketing, file store
  • Sem código de integração custom para cada serviço novo
Hooks

Injete comportamento no ciclo de vida

  • preToolUse, antes de qualquer tool, valida e pode bloquear
  • postToolUse, depois da tool, cache, redação, telemetria
  • onError, em falha de tool ou agent, recuperação e alerta
  • Onde governança, log e política realmente vivem
Boas práticas

Oito práticas que separam bibliotecas de cemitérios.

01Responsabilidade única. Uma skill, um trabalho. Se faz duas coisas, divida.
02Descrição primeiro. Gaste mais tempo na descrição do que no corpo. A descrição é o que ganha a carga.
03Carga preguiçosa de referências. SKILL.md fica abaixo de 200 linhas. Detalhe vai em referências, abertas sob demanda.
04Versionada em git. Skills são software. Trate o changelog a sério.
05Testada em CI. Um prompt de teste por caso de uso, snapshot, rodando em todo PR que toca skills/.
06Política de recusa explícita. Diga em voz alta o que a skill não fará. Economiza retentativas a jusante.
07Template de saída explícito. Drift de formato é caro. Pregue o formato de saída, não deixe o LLM improvisar.
08Cure, não acumule. Apague skills sem uso. Um cemitério de skills mortas é pior que nenhuma skill.
Glossário

Cinco termos. Um vocabulário comum.

Termo Definição
AgentRuntime. LLM mais instruções mais ferramentas. Mantém a conversa, escolhe tools, produz saída.
