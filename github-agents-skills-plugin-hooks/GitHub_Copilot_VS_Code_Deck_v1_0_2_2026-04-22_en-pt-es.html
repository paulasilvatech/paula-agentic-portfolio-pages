Part I
I

The paradigm shift.

Why GitHub Copilot stopped being autocomplete the day Chat modes arrived.

Before vs after

GitHub Copilot is no longer one thing. It is three modes you switch between on purpose.

2022 - 2023, autocomplete era

One mode. Ghost text.

  • Inline suggestions based on the cursor position.
  • Accept, reject, cycle with Alt+]. That was it.
  • The developer keeps the plan in their head.
  • No memory, no files, no terminal, no tools.
2025 - 2026, chat modes era

Three modes. One panel. A real collaborator.

  • Ask, Plan and Agent, each with a different contract.
  • Reads the repo, runs the terminal, edits many files.
  • The plan becomes an artifact you can review and version.
  • Configured by instructions, chatmodes, prompts and MCP servers.
Copilot Chat modes announcement and evolution github.blog/changelog · code.visualstudio.com/docs/copilot
The three modes

Ask, Plan, Agent. Three tools. One keyboard shortcut.

Mode 01 · Ask

Ask

When you need an answer, not a change.

Conversational Q&A grounded in the open repo. Reads files, references symbols, explains code. Never writes to disk.

Artifact → a reply in chat

Mode 02 · Plan

Plan

When the change is bigger than a single file.

Produces a structured plan before any code is written. Asks clarifying questions, lists affected files, proposes a sequence of steps.

Artifact → a plan.md you can review

Mode 03 · Agent

Agent

When you want the change executed, not described.

Autonomous loop that edits files, runs terminals, fixes errors and verifies its own work against tests you trust.

Artifact → a working diff

The same Copilot Chat panel. One dropdown switches the mode. The mental model you use for each is what changes.

A useful analogy

Think of the three modes as three colleagues with very different jobs.

Ask is like
A senior colleague on Slack
You paste a snippet, ask a question, get an answer in 30 seconds. No commits, no branches, no side effects. Fast.
Plan is like
A tech lead at the whiteboard
Before anyone touches the code, you agree on what will change, where, in what order. The plan is the contract.
Agent is like
A contractor doing the work
You handed the plan to someone. They go, do, come back with a PR. Your job is to review, not to type.

The mistake most teams make is sending every task to the contractor. If you do that for a question, you will wait 4 minutes for a 10-second answer.

Six prompt patterns for Ask

The shape of the prompt decides the shape of the answer.

01 · Explain
"Explain this function at a senior level."
Use before touching code you did not write. Forces a mental model before a change.
02 · Why
"Why does this code do X and not Y?"
Surfaces assumptions embedded in the code. Often reveals a trade-off the author never wrote down.
03 · Risk
"What breaks if concurrency doubles?"
Forces a non-happy-path review. Catches race conditions and shared-state bugs before they ship.
04 · Compare
"How does this differ from invoice.service?"
Uses two files as context. Great for spotting inconsistencies across a codebase.
05 · Teach
"Explain this as if I am new to the team."
Great onboarding loop. Asks Copilot to translate from expert to novice vocabulary.
06 · Trace
"Trace the call path from handler to db."
Chains multiple files. Perfect before refactoring an entry-point.
Prompt patterns validated in arXiv 2501.13282 · arXiv 2602.03593
Ask, best practices

A good Ask prompt is a question with scope. A bad one is a wish.

Weak prompt

"Help me with this code."

  • No goal, no scope, no success criterion.
  • Copilot guesses what help means. Usually wrong.
  • Answer is generic and you lose 3 minutes pasting more context.
Strong prompt

"Explain how processPayment handles Stripe timeouts. If it doesn't retry, suggest a safe idempotency pattern."

  • Goal is explicit, scope is one file, criterion is a pattern.
  • Answer is immediately actionable.
  • Works even on a large codebase because the target is named.

Plan mode.

Before any line of code, a contract that fits on one screen.

Plan mode, live

A request becomes a plan. The plan is the artifact.

· src/webhooks/stripe.router.ts, wire middleware before handler.
· test/stripe.webhook.spec.ts, 4 cases (valid, missing header, tampered body, expired timestamp).

Out of scope: refactoring existing handler logic, retry policy.
Risks: clock skew on timestamp tolerance. Default 300s, confirm.

The four stages of Plan

Plan mode runs a four-step loop. You stay in the driver's seat.

The whole loop costs 2 to 4 minutes for a medium-sized change. Worth every second compared to an agent that starts coding on a wrong assumption.

When to use Plan

Plan mode pays off when the cost of the wrong change is higher than the cost of thinking first.

Use Plan when
The change touches more than 2 files · a public contract (API, DB schema, event) is involved · the task has security or compliance implications · you are the new engineer on that module · the team needs a written record of what was decided.
Skip Plan when
You're renaming a variable, fixing a typo, updating a one-line config, or writing a throwaway script. The overhead outweighs the benefit. Use Ask or go straight to Agent.
Part IV
IV

Agent mode.

Multi-step execution inside the editor. With guardrails, not without them.

The agent loop

Observe → Think → Act → Verify → Iterate. Every cycle gets closer to green.

Unlike Ask (one turn) and Plan (four stages with you in control), Agent owns the loop. Its job is to keep iterating until the verifier says green, or until a guardrail stops it.

Agent mode, live

The agent edits files, runs the terminal, and comes back with a diff.

Agent guardrails

The difference between "demo" and "production" is four guardrails.

Guardrail 01
Terminal allowlist
Explicit list of commands the agent may run unattended. npm test yes, rm -rf never.
Guardrail 02
Per-edit diff review
Accept or reject each file change. The agent pauses on ambiguity, the human decides.
Guardrail 03
Stop-loss criteria
Written in your instructions: max 10 iterations, stop on any test you don't own, ping the human.
Guardrail 04
Branch isolation
Agents work on a disposable branch. If the loop drifts, you throw it away and try again.
Guardrail taxonomy based on arXiv 2509.14745 · Anthropic, Building Effective Agents · docs.github.com/copilot

Customizing.

Instructions, chatmodes, prompts, models and MCP. The four layers that make Copilot yours.

The configuration stack

Four files (and one folder) turn Copilot into your team's Copilot.

.github/copilot-instructions.md

Repo-wide instructions

The baseline rules every mode reads: coding standards, tech stack, testing expectations, what "done" means on this repo.

Scope · every conversation in this repo
.github/chatmodes/*.chatmode.md

Custom chat modes

A reusable persona with its own system prompt, toolset, and model pick. Appears as a mode option next to Ask / Plan / Agent.

Scope · one conversation, one job
.github/prompts/*.prompt.md

Reusable prompts

A named, parameterized prompt you invoke from the chat with /prompt-name. Perfect for "write a migration", "review this PR".

Scope · one invocation, one prompt
.vscode/mcp.json

MCP server config

Declares the MCP servers available to Agent mode. Each server adds a set of typed tools the agent can call with consent.

Scope · agent mode, per workspace
Example, copilot-instructions.md

A good instructions file is short, opinionated, and testable.

.github/copilot-instructions.md
# Payment Service, GitHub Copilot Instructions

## Stack
- Node 20, TypeScript 5.4, NestJS, PostgreSQL 16
- Tests, Vitest. Never suggest Jest or Mocha.
- Logging, pino with requestId. Never console.log.

## Rules
- Always use async / await. No .then chains.
- Every handler returns a typed Result<T, Error>, never throws.
- For any Stripe call, include an idempotency_key.
- Never log card numbers, tokens, or secrets. Use redact() helper.

## Done means
- New file or edit is covered by a Vitest case in /test.
- npm run lint passes. npm run typecheck passes.
- No TODO left pointing to unowned code.

## Escalate to a human when
- The change touches the database schema or a public API.
- A test we don't own (/test/vendor) fails.
- The diff exceeds 400 lines without a companion plan.md.

Common mistakes

Four traps I have seen in every team adopting GitHub Copilot.

TRAP 01Using Agent mode for questions. You wait 4 minutes for what Ask would answer in 15 seconds.
TRAP 02Skipping the plan. The agent loop starts on a wrong assumption, iterates in the wrong direction, and burns tokens.
TRAP 03No copilot-instructions.md. Every engineer gets a different Copilot. Style drifts. Reviews become exhausting.
TRAP 04Giving Agent a terminal with no allowlist. It's not a question of if, it's when something destructive runs.
Monday morning

Five actions to take in the first hour you're back at your desk.

ACTION 01Create .github/copilot-instructions.md in your top-repo. Ten lines is enough to start.
ACTION 02For your next task, try all three modes on purpose. Notice which one fits.
ACTION 03Write one chatmode for the role you repeat most (reviewer, tester, migrator).
ACTION 04Configure one MCP server, GitHub is the easiest win, and use it with Agent.
ACTION 05Set a stop-loss rule in your instructions. Max iterations, unowned tests, diff size.
