Part I · Two agent types

Two types of agent. One fundamental difference in tool access and identity.

Generalist agent

Workspace tools, loaded skills

  • Default orchestrator, handles 70-80% of tasks
  • No dedicated .agent.md identity file
  • Augmented by SKILL.md when domain knowledge needed
  • Tools are workspace-level and shared
  • Appropriate for domain knowledge tasks (Shape A)
Custom agent

Exclusive tools, defined identity, bounded scope

  • Defined by .agent.md: identity, tone, tool list, scope
  • Dedicated MCP server connections, not shared
  • Bounded domain: security, testing, infra, docs
  • Receives tasks via handoff, returns typed artifacts
  • Appropriate for specialized identity tasks (Shape B)

The generalist handles 70-80% of tasks in a mature workspace. Custom agents handle the 20-30% requiring exclusive tool access, distinct identity, or multi-stage handoffs.

Part II · Skills

Anatomy, lifecycle, and the five-section structure.

A SKILL.md is not a prompt, not a config, not a script. It is a domain-specific knowledge artifact with a lifecycle, a maturity model, and three activation modes. This part covers everything needed to build, ship, and maintain a skill that actually holds quality over time.

Part II · SKILL.md anatomy

Five sections. Every skill has all five. No exceptions.

01
Description + SFA test
YAML frontmatter: name, description, globs, alwaysApply. Scope, Format, Audience must be explicit.
02
Domain rules
Non-negotiable constraints, conventions, naming, anti-patterns. The hard knowledge of the domain.
03
Step-by-step workflow
Numbered sequential steps explicit enough to be reproducible. References to tools and scripts.
04
Output templates
Concrete examples: format, structure, file naming. The agent matches templates, not imagination.
05
Validation checklist
Pass/fail criteria applied before delivery. If any item fails, the agent revises. Quality gate built into the skill.
SKILL.md · docx skill example
---
name: docx
description: |
  Create Word documents (.docx) with
  professional formatting. Triggers on:
  Word doc, .docx, report, memo.
globs: ["**/*.docx"]
alwaysApply: false
---

# 01 · Description
## SFA: Scope / Format / Audience
Scope: create .docx files...
Format: python-docx output...
Audience: generalist agent...

# 02 · Domain rules
NEVER use Arial for body text.
ALWAYS version files with date.

# 03 · Workflow
1. Read skill file via view tool
2. Install python-docx if missing
3. Run the generation script...

# 04 · Output templates
{Title}_v{version}_{date}_{locale}.docx

# 05 · Validation checklist
- [ ] File opens without error
- [ ] All headings use correct style
- [ ] File copied to /outputs/
Part II · Skill lifecycle

Three phases, seven steps. From idea to distributed, production-ready skill.

Phase 01 · Essential
Every skill needs these three
01Identify: purpose, SFA test, naming convention, folder structure.
02Create SKILL.md: YAML frontmatter, all five sections, concrete output examples.
03Distribution scope: workspace-wide, folder-scoped, or project-specific. Configure applyTo and globs.
Phase 02 · Optional
Add power and references
04Automation scripts: Python, Bash, or Node.js. Shebangs, arg validation, error handling.
05Enrich: external docs, API links, example outputs, pre-built templates, data files.
Phase 03 · Quality
Validate before distribution
06Test end-to-end with real prompts. Every section must produce expected output. Run all five checklist items.
07Distribute: publish to scope, validate glob match discovery, confirm another agent can invoke it.

Level 1 is ~40 lines. Level 5 includes scripts, CI validation, and automated tests. Most enterprise skills operate well at level 3.

Part II · Skill maturity model

Five levels. Start at 1. Ship at 3. Automate to 5.

01
Basic
SKILL.md body only. ~40 lines. Covers description and workflow. No templates, no checklist. Usable but fragile.
02
Structured
All five sections. References and output templates complete. Checklist operational. Ready for single-user production.
03
Enriched
Scripts, portal references, enriched content. Multiple examples. Glob patterns tuned. Team distribution ready. Most enterprise skills live here.
04
Validated
Automated tests, CI gate. Cannot ship unless tests pass. Intent debt tracked weekly. Quality regression detectable.
05
Production
Full pipeline. Version-pinned releases. Cross-workspace registry. Automated promotion via CI. The skill as first-class asset.

Most teams start with level 1 skills that work. The cost comes when level 1 skills proliferate without promotion: inconsistent outputs, duplicate domain knowledge, no quality gate.

Part II · Three activation modes

The agent does not guess. It matches one of three modes.

Three modes, three trade-offs. Glob match is efficient. alwaysApply is powerful but costs tokens on every call. Explicit invoke is precise but requires routing awareness. Most skills use one mode only.

Part III · Custom agents

Identity, tools, scope, and the .agent.md contract.

A custom agent is not a named chatbot. It is a software component with a defined identity, exclusive tool access, a bounded scope, and a structured handoff protocol. This part covers the .agent.md file, four concrete agent examples, and when to build versus when to use a skill.

Part III · .agent.md anatomy

Six sections define a custom agent. The .agent.md is the contract with the orchestrator.

01
Identity and persona
Name, role, tone, domain. Who this agent is, not just what it does.
02
Tool manifest
Which MCP servers, CLIs, and APIs this agent has exclusive access to. No shared-by-default.
03
Scope and boundaries
What this agent handles. What it explicitly refuses. Prevents scope creep.
04
Input / output contracts
What handoff payload the agent accepts. What typed artifact it returns.
05
Skills loaded
Which SKILL.md files this agent loads. Custom agents also use skills for domain knowledge.
06
Escalation rules
When to return to orchestrator without completing. Which conditions require human review.
.agent.md · security-agent
---
name: security-agent
version: 1.0.0

# 01 Identity
identity:
  role: Security reviewer
  tone: rigorous, evidence-based
  domain: application security

# 02 Tool manifest (exclusive)
tools:
  - vault-mcp      # secrets access
  - semgrep-cli    # static analysis
  - snyk-api       # dependency scan

# 03 Scope
handles: [security review, secrets audit]
refuses: [feature implementation]

# 04 I/O contracts
input: code + tests + plan
output: security-report.md (CVSS scores)

# 05 Skills
skills: [security/SKILL.md]

# 06 Escalation
escalate_on:
  - critical CVE found
  - credential exposure detected
---
Part III · Four real agents

Four agents. Each one exists because no skill alone was enough.

Security agent
Vault + SAST

Exclusive: HashiCorp Vault MCP, Semgrep CLI, Snyk API. Rigorous, evidence-based tone. Produces security-report.md with CVSS scores. Escalates on critical CVE.

vault-mcp · semgrep · snyk
Test agent
CI/CD integration

Exclusive: GitHub Actions MCP, Jest runner, coverage tool. Generates test suites from acceptance criteria. Executes them, returns results. Escalates on flaky tests.

actions-mcp · jest · coverage
Modernization agent
IaC + migration

Exclusive: Terraform MCP, Azure SDK. Transforms legacy infra to IaC. Produces migration plan and diff. Escalates on data loss risk. No security audit.

terraform-mcp · azure-sdk
Documentation agent
Docs + publishing

Exclusive: Confluence MCP, brand style guide. Friendly, clear tone. Generates and publishes structured documentation. No code review, no security gates.

confluence-mcp · brand-skill

Each agent handles one bounded domain. The orchestrator knows which to call based on the task type detected in routing. Boundaries are explicit, not emergent. The ratio in a healthy workspace: ~1 custom agent per 8-12 skills.

Part IV · Orchestration

Four-phase flow, decision tree, handoff sequence.

Orchestration is the runtime implementation. Every request passes four phases. The routing decision is made once, in Phase 2, by applying the four-question decision tree. Handoffs follow a structured protocol: the orchestrator is always the hub.

Part IV · Four-phase orchestration

Every request passes four phases. Routing decided once, in Phase 2.

Phase 01
Entry
  • User request received by orchestrator
  • Orchestrator loads its own .agent.md identity
  • Request classified: task type detected
Input
Phase 02
Analysis + routing
  • Decision tree evaluated against task type
  • Skill search: does SKILL.md exist?
  • Path A: custom agent + skill, or Path B: generalist + skill
Routing
Phase 03
Execution
  • Skills loaded, domain knowledge injected
  • .instructions.md applied automatically
  • Task executed with full context
Execute
Phase 04
Validation + delivery
  • Output validated against skill checklist
  • Quality gate: format, content, conventions
  • Final response delivered to user
Output

The orchestrator is the only agent the user interacts with directly. All sub-agents operate behind it. Every handoff returns to orchestrator. Agents never communicate directly with each other.

Part IV · Decision tree

Four yes/no questions. One of three outcomes. No ambiguity.

Does the task require exclusive tool access? (MCP, API, CLI)
YESCustom agent + skill. No generalist agent has exclusive tool access.
NOContinue to next question.
Does the task require a distinct identity or tone? (e.g. rigorous security)
YESCustom agent + skill. Identity is in .agent.md, not improvised per call.
NOContinue to next question.
Is this a multi-stage pipeline with handoffs between agents?
YESCustom agent pipeline. Each stage needs a bounded agent to maintain output contract.
NOContinue to final question.
Does a SKILL.md exist that covers this domain?
YESGeneralist agent + skill. Load it and proceed.
NOGeneralist only. Or: create the skill first.

Part IV · Multi-agent handoff · How work flows from agent to agent

Four handoffs. One orchestrator. Zero direct agent-to-agent calls.

Context accumulates at every handoff. Security receives code + tests + plan. No handoff loses prior work. The orchestrator is hub; no agent shortcuts to another.

Part V · Anti-patterns

Four anti-patterns you will meet before any best practice.

Anti-pattern 01
Skill-as-megaprompt
A 2,000-line SKILL.md covering everything. No clear scope. Activates on every glob. Token overhead per call exceeds the value of the knowledge. Fix: split into three targeted skills, each passing the SFA test.
Anti-pattern 02
Agent-for-everything
A single custom agent with all permissions. No bounded scope. No exclusive tool rationale. First incident compromises the entire workspace. RBAC and scope boundaries are non-negotiable.
Anti-pattern 03
Skills never versioned
SKILL.md files edited in place without version control. Output quality changes silently. No rollback. No change log. Treat every SKILL.md as code: version, review, test, promote through the maturity levels.
Anti-pattern 04
Routing by intuition
Per-call decisions based on feel. No decision tree, no routing logic. Inconsistent results, inconsistent cost. No measurement baseline. One hour to implement the four-question tree; no budget needed.

Anti-patterns 01 and 02 are the most costly. Megaprompt skills inflate token spend silently. Agent-for-everything creates security surface risk. Both are solved by the SFA test and decision tree applied early.

Part V · Workspace diagnostic

Six diagnostic questions. Any "no" is a gap worth closing.

Q01
Do you know every skill active in your workspace right now?
If no: run a skills inventory. Unknown skills with alwaysApply true are a silent token drain and a quality risk.
Q02
Do all custom agents have a .agent.md with an explicit tool list?
If no: any agent using shared tools is a custom agent in name only. The tool manifest is the contract. Without it, scope is undefined.
Q03
Can you trace which skill was loaded for any given task output?
If no: you cannot debug quality regressions. Skill activation must be logged. No observability means no quality control.
Q04
Is there a routing decision tree, or does each team member decide ad hoc?
If ad hoc: the four-question tree takes one hour to implement. Without it, agent and skill usage is inconsistent and unmeasurable.
Q05
Are all skills version-controlled and reviewed before distribution?
If no: treat every SKILL.md as code. The lifecycle in Part II defines the promotion path from draft to production.
Q06
Is the ratio of custom agents to skills in a healthy range (1:8 or better)?
If no: you are either over-engineering (too many agents) or under-specializing (too few skills). Both signal misrouting patterns.

Five things you can do Monday with no new budget.

01 · AUDITInventory every skill in your workspace. Note which have alwaysApply true, their line counts, and whether they have a validation checklist.
02 · DEFINEApply the SFA test to your three most-used skills. Scope, Format, Audience. Document any gap before the next team session.
03 · ROUTEPrint the four-question decision tree. Apply it explicitly to the next five task requests your team receives. Track routing decision and actual outcome.
04 · CONTRACTReview your custom agents' .agent.md files. Does each have an explicit tool list, a scope boundary, and an escalation rule? If not, add what is missing this week.
05 · MEASURESchedule a 30-day review. One metric: what percentage of tasks were routed through the decision tree? Any number above zero is progress.
